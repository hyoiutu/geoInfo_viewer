# Issueレビュースキル (issue-review)

このスキルは、AIエージェントがGitHub Issueの実装に着手する前に、Issueの内容を事前レビューするための指示書です。[issue-implement](../issue-implement/SKILL.md)スキルの一部として組み込まれており、単独では基本的に呼び出しません。

これまでの運用では、Issueの本文だけでは分からない懸念（外部サービスの認証情報が無い、分割基準が曖昧、UI/UX要望の裏取りが不足している等）が実装の途中や実装後のレビューで初めて発覚し、手戻りが発生していました。過去に発覚した懸念のパターンを[issue_review_notes.md](../../../issue_review_notes.md)に蓄積し、同種の懸念を実装着手前に検出できるようにします。

## 🎯 起動条件

- issue-implementスキルの手順1（Issue一覧・詳細の取得）の直後、実装着手（対話モードの確認・自律モードのブランチ作成）前に必ず実行する。
- ユーザーから「Issue #<番号>をレビューして」等、本スキル単独での実行を明示的に指示された場合。

---

## 📋 実行プロセス（手順）

### 1. レビュー観点の読み込み
- [issue_review_notes.md](../../../issue_review_notes.md)を読み込み、蓄積されたレビュー観点の一覧を把握する。

### 2. 対象Issueとの照合
- 対象Issueの本文を、読み込んだ各観点と照らし合わせる（issue-implementから呼ばれた場合、既に取得済みのIssue詳細を再利用してよい）。
- 現時点で蓄積されている主な観点（詳細は[issue_review_notes.md](../../../issue_review_notes.md)参照）:
  1. 外部サービス連携に必要な認証情報が用意されているか
  2. 大規模な再編・分割Issueは、境界の判断基準を先に固める必要が無いか
  3. 色・レイアウト等の主観的なUI/UX要望は、設計原則での裏取りが必要ないか
  4. スタック構成のPRになる見込みがある場合、finish-reviewのbase整合性チェックの対象になることを意識しているか

### 3. 懸念への対応
懸念が見つかった場合、以下のように対応する。

- **対話モード**: `AskUserQuestion`等でユーザーに確認し、実装に進む前に解消する。
- **自律モード**:
  - 実装不可能な明確なブロッカー（認証情報が無い等）の場合、そのIssueをスキップする。[issue-implement](../issue-implement/SKILL.md)スキル手順6「行き詰まった場合」に従い、中断理由をプロジェクトメモリに記録した上で次のIssueに進む。
  - 実装は可能だが判断が必要な曖昧さの場合、最も妥当な方針を選んで実装を進める。[issue-implement](../issue-implement/SKILL.md)スキル手順4の既存ルール（コード上に一時コメントを残さず、PR作成後にGitHubのインラインレビューコメント機能で代替案と選定理由を提示する）に従う。
- 懸念が見つからない場合は、その旨を記録した上でそのままissue-implementスキルの後続手順（対話モード2-A / 自律モード2-B以降）へ進む。

### 4. ノウハウの蓄積
- レビューの結果、[issue_review_notes.md](../../../issue_review_notes.md)に載っていない新しい種類の懸念（実装中・実装後のレビューで初めて発覚した手戻りの原因）が見つかった場合、対応するIssue/PR番号を実例として添えて観点を追記する。

---

## 🚫 注意事項
- 本スキルはIssueの内容を判断するのみで、コード変更は行わない。
- 本スキル自体（観点のカテゴリ分け等の構造）を変更する場合は、他のルールファイル（rules.md等）と同様、人間からの指摘を受けて更新する。日々の運用で追記するのは[issue_review_notes.md](../../../issue_review_notes.md)側の個別観点であり、本ファイルの構造自体を都度書き換えるものではない。
