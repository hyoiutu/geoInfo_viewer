# レビュー完了後処理スキル (finish-review)

このスキルは、AIエージェントがPRレビューコメントへの対応（[pr-review-respond](../pr-review-respond/SKILL.md)等）を終えた後、push・PRマージ・次にレビューするPRの選定・そのPRブランチへの祖先ブランチの取り込みまでを一貫して行うための指示書です。

これまでの運用では、あるPR（例: PR A）がマージされると、Aから分岐した子孫ブランチ（PR B、さらにBから分岐したPR C…）全てで、次にレビューを始める前に手動で`main`をマージしCHANGELOG.mdの競合を1世代ずつ解消する必要がありました。世代が深いほどこの作業が繰り返し発生し負担になっていたため、この手順を自動化します。

## 🎯 起動条件

- ユーザーから「finish-reviewして」「レビュー終わったので次の準備をして」等の明示的な指示があった場合、または本スキルが明示的に呼び出された場合。
- 本スキルは常に対話モードで実行する。マージという不可逆性の高い操作を伴うため、ユーザー不在での自律的な実行は行わない。

---

## 📋 実行プロセス（手順）

### 1. 現在のブランチのpush
- カレントブランチに未pushのコミットがあれば、`git push -u origin <ブランチ名>`でpushする。**force push（`--force`/`-f`）は絶対に使用しない。**
- 既にリモートと同期済みで差分が無い場合は本手順をスキップしてよい。

### 2. 対応するPRのマージ（要ユーザー確認）
- GitHub MCP（`list_pull_requests`、`head: <owner>:<ブランチ名>`）で、カレントブランチに対応するOPENなPRを検索する。
- PRが見つからない場合は、その旨をユーザーに伝えてスキルを終了する（マージ対象が無いため）。
- **PRの`base`が`main`以外の場合、マージを実行する前に必ず以下の手順でbaseの整合性を確認・修正する。**
  - 背景: `base`がstaleなfeatureブランチのままマージすると、本来`main`へ入るべき内容がそのfeatureブランチへ紛れ込み、後続の子孫PRを経由して初めて（本来のPRを経由せず）`main`へ混入する事故につながる（実際に発生した事例をCHANGELOG.md「finish-reviewスキルのPRマージ手順にbase整合性チェックを追加した」に記録）。
  1. `base`ブランチに対応するPRを検索する（`list_pull_requests`、`head: <owner>:<baseブランチ名>`）。
  2. そのPRが既にマージ済み（closed & merged）であれば、`base`ブランチの内容は実質`main`に吸収されているとみなし、`update_pull_request`で対象PRの`base`を`main`へ変更する。変更後、対象PRを`pull_request_read`（`get`）で再取得し、`mergeable_state`が`clean`であることを確認する（`dirty`等の場合はコンフリクトが残っているため、手順4と同様の要領で対象ブランチへ`main`をマージしコンフリクトを解消してから再度確認する）。
  3. そのPRがまだOPENである場合、**対象PRをこの時点でマージしてはならない**。祖先PR（`base`ブランチに対応するPR）が先にマージされる必要がある。その旨をユーザーに伝え、先に祖先PRをレビュー・マージするか、対象PRの扱いをどうするか判断を仰いだ上でスキルを中断する。
  4. `base`ブランチに対応するPRが見つからない場合（独立したブランチ等、想定外の状態）も、判断に自信が持てないため処理を中断しユーザーに確認する。
- PRが見つかった場合（`base`が`main`である、または上記手順で`main`へ修正済みの場合）、マージを実行する**前に必ず**`AskUserQuestion`で以下を提示し、Y/N形式の確認を得ること。この確認は省略してはならない。
  - PR名（番号・タイトル）
  - マージするブランチ（head）
  - マージ先ブランチ（base。常に`main`であることを確認した上で提示する）
- ユーザーが承諾した場合のみ、`merge_pull_request`でマージを実行する（マージ方法は特別な指定が無い限りリポジトリの標準的な方法、通常は`merge`を使う）。
- ユーザーが拒否した場合は、マージを行わずにその旨を確認した上で手順3へ進むかスキルを終了するかをユーザーに確認する。
- **マージ成功後、対応するGitHub Issueをクローズする**: PRのタイトル・ブランチ名（`<type>/issue-<番号>-...`）から対象Issue番号を特定し、`issue_write`で`state: closed`・`state_reason: completed`に更新する。マージ済みPRへの参照を添えたコメントも残す。対応するIssue番号が特定できない場合（Issueに紐付かないPRの場合）はこの手順をスキップする。
  - 背景: マージ済みのPRに対応するIssueがクローズされずOPENのまま残ってしまい、`issue-implement`スキルの自律モードが次の対象Issueを誤って選定しかけた事例があったため追加した（2026-07-14）。

### 3. 次にレビューするPRの選定
- GitHub MCP（`list_pull_requests`、`state: open`）でOPENなPR一覧を取得する。
- `AskUserQuestion`で、取得したPR一覧（番号・タイトル・ブランチ名）を選択肢として提示する。**「レビューをやめる」という選択肢も必ず含めること。**
- ユーザーが「レビューをやめる」を選んだ場合、ここでスキルを終了する。
- ユーザーが特定のPRを選んだ場合、手順4へ進む。

### 4. 選択したPRのレビュー準備（祖先ブランチの取り込み）
- `git fetch origin`でリモートの最新状態を取得する。
- 選択したPRの対象ブランチ（head）から`main`までの祖先チェーンを求める。
  - 選択したPRの`base`ブランチを確認する。`base`が`main`であれば、そこでチェーンの探索は終わり。
  - `base`が`main`でない場合、その`base`ブランチに対応するPRを検索する。**そのPRが既にマージ済み（closed & merged）であれば、そのブランチの内容は実質`main`に吸収されているとみなし、遡りをそこで打ち切ってよい（`main`を直接の取り込み元として扱う）。まだOPENであれば、さらにそのPRの`base`を再帰的に遡る。**
- 求めたチェーンの**`main`に近い方から順に**、「1つ祖先側のブランチ（無ければ`main`）の内容を、対象ブランチへ`git merge`で取り込む」を繰り返し、最終的に選択したPRの対象ブランチまで到達させる。
  - 各ステップ: `git checkout <ブランチ名>` → `git merge <祖先ブランチ>`。
  - **コンフリクトが発生した場合の解決手順:**
    - **CHANGELOG.mdの競合**: これまでの運用実績上、追記どうしの競合であることが多い。両方のエントリを保持した上で、ブランチのトポロジー順ではなく**各エントリが実際に作業された時系列順**に並べ替えて解決する（該当エントリの日付・コミット履歴から実際の順序を判断すること）。
    - **上記以外の競合**は、次の基準を両方満たす場合に限りエージェントが自律的に解決してよい。
      1. 解決後の内容が[specs/system_specification.md](../../../specs/system_specification.md)・[designs/以下の設計書](../../../designs/)の記述と矛盾しないこと。
      2. 解決後の内容がプロジェクトのルール（[AGENTS.md](../../../AGENTS.md)・[typescript_rules.md](../../../typescript_rules.md)・[react_rules.md](../../../react_rules.md)・[comment_rules.md](../../../comment_rules.md)・[design_principles.md](../../../design_principles.md)・[ui_rules.md](../../../ui_rules.md)・[test_rules.md](../../../test_rules.md)・[branch_rules.md](../../../branch_rules.md)・[commit_rules.md](../../../commit_rules.md)）に違反しないこと。
    - **`specs/system_specification.md`・`designs/`以下の設計書自体の記述内容が競合している場合（仕様書・設計書変更レベルのコンフリクト）**は、上記基準を満たすかどうかに関わらず必ず処理を中断し、競合している内容を要約してユーザーに判断を求めること。仕様・設計の解釈・優先順位はエージェントが独断で決めてはならない。
    - **上記いずれの基準にも当てはまらない、判断に自信が持てない競合（アプリケーションコードの実質的な内容衝突等）**も、無理に解決せず処理を中断し、競合しているファイル・箇所を要約してユーザーに判断を求めること。ユーザーの指示を受けてから解決を再開する。
  - コンフリクト解決後の`git commit`は、通常のマージコミットの確定操作であり[auto-commitスキル](../auto-commit/SKILL.md)の対象外（マージコミットにはスキルを介さず通常通り`git commit`してよい）。ただし、コンフリクトが無く自動マージされた場合も含め、各マージ成功後は[commit_rules.md](../../../commit_rules.md)記載の検証コマンド（テスト・lint・型チェック等）を実行し、全て成功することを確認してから次のステップおよびpushへ進むこと。
  - 各ステップのマージ・検証が完了したら、`git push`でそのブランチの更新をリモートへ反映する（後続ステップの`git fetch`や、次回本スキルを実行する際にリモートの状態と一致させておくため）。
- 最終的に選択したPRの対象ブランチが、祖先チェーン全体（＝最新の`main`）の内容を取り込んだ状態になったら、その旨をユーザーに伝えてスキルを終了する。レビューの実施自体（コメントの読解・返信等）は[pr-review-respond](../pr-review-respond/SKILL.md)の範囲であり、本スキルの対象外。

---

## 🚫 注意事項
- 本スキルはPRのマージ・複数ブランチへの`git push`という影響範囲の大きい操作を伴う。**手順2のマージ前確認（Y/N）は絶対に省略しないこと。** 他の手順（push・軽微な競合の機械的解決）はユーザーの逐次確認を必要としないが、確認無く進めてよいのはマージの承認のみである。
- **手順2のbase整合性チェックも省略しないこと。** `base`が`main`以外のままマージすると、内容が`main`以外のブランチへ紛れ込む事故になる。
- **`git push --force`/`-f`は、いかなる場合も絶対に禁止。** このルールに例外は無い。
- 競合解決の判断に自信が持てない場合は、必ず処理を中断しユーザーに判断を委ねること。誤った判断による競合解決は複数のリモートブランチへ連鎖的に反映されるため、慎重に扱うこと。
