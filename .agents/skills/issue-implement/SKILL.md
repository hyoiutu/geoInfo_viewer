# AI Issue実装スキル (issue-implement)

このスキルは、AIエージェントがGitHub Issueの内容をもとに実装・修正を進めるための指示書です。**対話モード**（ユーザーと都度やり取りしながら1件のIssueを進める）と**自律モード**（ユーザー不在で複数Issueを連続して消化する）の2種類の実行モードを持ちます。

## 🎯 起動条件

- ユーザーから「Issue #<番号>を実装して」「GitHub Issueを対応して」等の指示があった場合、または本スキルが明示的に呼び出された場合 → **対話モード**。
- `/loop`スキル経由で本スキルが繰り返し呼び出された場合、またはユーザーから「Issueを自律的に消化していって」「離席するので進めておいて」等、複数Issueの連続処理・都度確認不要の実行を明示的に指示された場合 → **自律モード**。

判定に迷う場合は対話モードとして扱い、ユーザーに確認すること。

---

## 📋 実行プロセス（手順）

### 1. Issue一覧・詳細の取得
- GitHub MCPサーバー（`list_issues`等）を用いて、対象リポジトリのIssue一覧（特にOPEN状態のもの）を取得します。
- 対象Issue番号が既に分かっている場合も、内容を正確に把握するため該当Issueの詳細を取得します。

### 2-A. 【対話モード】ユーザーへの確認（AskUserQuestion）
- 実装（および後述のブランチ作成）を開始する前に、**必ず**`AskUserQuestion`等を用いて以下の3点をユーザーに確認してください。この確認を省略して実装やブランチ作成を進めてはいけません。
  1. **対象Issue**: どのGitHub Issue（番号）を対象に実装を進めるか。
  2. **ブランチ名**: ユーザーの希望があればそれを採用し、希望がなければ[branch_rules.md](../../../branch_rules.md)の命名規則（`<type>/issue-<Issue番号>-<内容>`）に従いAIが提案します。
  3. **派生元ブランチ**: デフォルトは現在のブランチです。ユーザーが別ブランチを希望すればそちらを採用します。

### 2-B. 【自律モード】対象Issue・ブランチ・派生元ブランチの自己決定
ユーザーへの確認を待てないため、以下の規則に従い自己判断で決定する。

- **対象Issue**: OPEN状態のIssueのうち、まだ対応済み/対応中でない最も番号の小さいものを選ぶ。
  - `git branch -a`と`gh pr list`で、[branch_rules.md](../../../branch_rules.md)の命名規則`<type>/issue-<番号>-...`に一致する既存ブランチ・PRを確認し、既にブランチまたはPRが存在するIssue番号は対象から除外する。
  - 該当するIssueが無くなった場合は、自律モードのループを終了する（後述手順7参照）。
- **ブランチ名**: [branch_rules.md](../../../branch_rules.md)の命名規則に従いAIが自動生成する。
- **派生元ブランチ**（「行き詰まったブランチの上には積まない」原則）:
  1. 直前に自律モードで着手し、正常にPRを作成できた（`gh pr list --head <ブランチ名>`でPRが存在する）Issueブランチがあれば、それを派生元とする（縦方向にブランチを連鎖させる）。
  2. 直前に着手したブランチが行き詰まって中断した場合（手順6参照）は、そのブランチではなく**その派生元**（さらに1つ前の正常なブランチ、無ければ元のカレントブランチ）を派生元とする。行き詰まったブランチの上に新しいブランチを重ねてはならない。
  3. 該当するブランチが無ければ、スキル起動時点のカレントブランチを派生元とする。
- カレントブランチに未コミットの変更が残っている場合、対話モードのようにユーザーへ確認することができないため、**auto-commitスキル経由でコミットしてから**（変更を破棄せず）ブランチを切る。

### 3. ブランチ運用規約の確認とブランチの用意
- [branch_rules.md](../../../branch_rules.md)を読み込み、「1 Issue = 1 Branch」の原則やブランチ命名規則、プッシュに関するルールを再確認します。
- 対象Issue用のブランチが既に存在する場合は新規作成せず、そのブランチをチェックアウトして再利用します（`git checkout <ブランチ名>`）。
- 存在しない場合は、手順2-A/2-Bで確認・決定した派生元ブランチから新規作成します（`git checkout -b <ブランチ名> <派生元ブランチ>`）。

### 4. 実装の実施（既存プロジェクトルールの遵守）
- 対象Issueの内容に基づき実装を行います。その際、AGENTS.md / rules.md / test_rules.md に定義された既存ルールを必ず遵守してください。
  - TDD（Red-Green-Refactor）に従って実装すること。
  - 実装・README.md・specs/以下の仕様書のいずれかに修正が生じた場合、3者間の乖離がないか確認し、乖離があれば連動して修正すること。
  - 上記の連動修正が発生した場合、CHANGELOG.mdに変更内容（何が変わり、実装・README.md・仕様書がそれぞれどう修正されたか）を記録すること。
  - コード規約はrules.mdに従うこと。
  - 単体テスト・E2Eテストの作成・実行はtest_rules.mdに従うこと。
- **【自律モード】実装方針でユーザーの判断が必要な分岐点に遭遇した場合**: 作業を止めてユーザーに確認を仰ぐのではなく、最も妥当と判断した方針を選択し、検討した他の選択肢・選んだ理由を明記した上でいったん実装を進める。この判断は手順7でPRの説明・コメントとしてユーザーに提示し、後から確認・修正できるようにする。
  - **PRの説明文にまとめて書くだけでなく、判断を行った具体的なコード箇所にも`// 設計判断（要確認）: <内容>`形式のインラインコメントを残すこと。** PRの説明文だけでは「具体的にコードのどこを指しているか分かりにくい」という問題があるため、レビュアーが該当行を見ればどの判断か即座に分かるようにする。このコメントは一時的なレビュー用マーカーであり、rules.mdの「タスクやPRへの参照を書かない」という通常のコメント規約の例外として扱う。レビューで判断が確定した後は、次にそのファイルを編集する際にコメントを削除するか、恒久的な設計意図の説明（rules.mdの通常のコメント規約に沿った形）へ書き改めること。

### 5. コミット
- 実装が完了し、コミット可能な差分が生じた場合は、必ず[auto-commitスキル](../auto-commit/SKILL.md)（[commit_rules.md](../../../commit_rules.md)）の手順を経由してコミットを行います。**本スキルおよびAIエージェントが直接`git commit`を実行してはいけません。**
- コミットの分割単位は、対話モード・自律モードとも「タスク（作業の意味のある単位）ごとに1コミット」とする。
- 対話モードではauto-commitスキルの通常の確認フロー（Y/N確認あり）に従う。**自律モードではauto-commitスキルの確認ステップ（Y/N）を省略してよいが、検証（テスト・lint・型チェック）は必ず全て実行し成功させること。**

### 6. 【自律モード】行き詰まった場合
- テストがGreenにならない、仕様が矛盾している、必要な認証情報・アクセス権が無い等、合理的な努力をしても解決できない場合、そのIssue・ブランチでの作業を中断する。
- 中断したブランチは**プッシュしない・PRを作成しない**（未完成の実装を公開しない）。
- 何が・なぜ行き詰まったか、どこまで進んだかをプロジェクトメモリに記録する。
- 次のIssueは、手順2-Bの派生元決定ルールに従い、行き詰まったブランチの上には積まず並行してブランチを切って続行する。

### 7. 【自律モード】プッシュとPR作成（実装完了後）
- 実装が完了し、全ての検証（テスト・lint・型チェック、UI変更があれば動作確認）が成功したら、通常のpush（`git push -u origin <ブランチ名>`）を行う。**force push（`--force`/`-f`）は絶対に使用しない。**
- 続けてPRを作成する（`create_pull_request`。リポジトリにPRテンプレートがあれば使用する）。
  - 手順4で自律的に判断したユーザー確認が必要な事項があれば、PRの説明文または最初のコメントとして明記し、代替案とその判断理由も添えて、ユーザーに選択を仰ぐ。
- PR作成後、`PushNotification`で簡潔にユーザーへ通知する（例:「PR #N作成: Issue #M対応完了」）。
- `/loop`経由で自律モードとして実行している場合は、続けて手順1に戻り次のIssueに着手する。対話モードで明示的に1件のみ依頼された場合は、この手順は行わずここで完了とする。

### 8. プッシュに関するルール（重要）
- 対話モードでは、ユーザーから明示的な指示が無い限りpushしない（従来通り、pushは人間が行う）。
- 自律モードでpush可能なのは、手順7（実装完了・全検証成功後の最終ステップ）のみ。実装途中や検証未完了の状態でpushしてはならない。
- **`git push --force`/`-f`は、対話モード・自律モードいずれであっても絶対に禁止。** このルールに例外は無い。
