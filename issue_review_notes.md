# Issue Review Notes（Issueレビューノウハウ）

本ファイルは、[issue-review](./.agents/skills/issue-review/SKILL.md)スキルが参照する、Issueの内容を実装前にレビューする際の観点を蓄積するドキュメントです。過去の実装・レビューで実際に手戻りや設計判断の負荷が発生した事例をもとに、同種の問題を実装着手前に検出できるようにします。

新しい教訓が得られた場合、末尾に観点を追加してください。各観点には、実例として対応するIssue/PR番号を残すこと。

---

## 観点一覧

### 1. 外部サービス連携に必要な認証情報が用意されているか
- **内容**: Issueが既存（Strava）以外の新しい外部API連携（Google Photos等）を要求する場合、対応する認証情報（`.env`/`.env.example`のクライアントID・シークレット・トークン等）が既に用意されているか確認する。
- **対応**:
  - 自律モード: 用意されていない場合、そのIssueはWIPラベル相当のブロッカーとして扱いスキップする（issue-implementスキル手順6「行き詰まった場合」に従う）。
  - 対話モード: 着手前にユーザーへ認証情報の準備状況を確認する。
- **実例**: Issue #23（写真閲覧機能、Google Photos連携）。`.env.example`には`STRAVA_*`のみ存在し`GOOGLE_*`系の変数が無いことを確認し、ユーザー判断で`WIP`ラベルが付与されスキップされた（2026-07-15）。

### 2. 大規模な再編・分割Issueは、境界の判断基準を先に固める
- **内容**: 「〇〇を分割する」「整理する」等、具体的な分割基準が本文に書かれていない大規模なリファクタ・ドキュメント再編Issueは、実装中に何度も個別の判断が必要になり手戻りが起きやすい。
- **対応**: 着手前に境界の判断基準を仮決めし、実装を進める。ユーザーの判断が必要な分岐点は、コード上のコメントではなくPR説明文・GitHubインラインコメントで代替案と選定理由を明記する（issue-implementスキル手順4の既存ルール通り）。
- **実例**: Issue #30（仕様書と設計書の分割基準）、Issue #29（クラス図の設置場所）。いずれもPRのインラインコメントで判断基準をユーザーに提示した。

### 3. 色・レイアウト等の主観的なUI/UX要望は、ユーザーの一次案も設計原則で裏取りする
- **内容**: ユーザー自身が具体的な色・配色案を提示した場合でも、それが必ずしも最終解とは限らない。実装後に実際の画面で確認して初めて「重なると見にくい」等の不十分さが判明することがある。
- **対応**: 実装前後で、dataviz skill等の確立された設計原則（色相・コントラスト・二次エンコーディング等）を参照し、可能であれば機械的な検証（`validate_palette.js`等）を行った上で提案する。ユーザーの一次案をそのまま実装して終わりにしない。
- **実例**: Issue #26（地図上の線の色）。初回実装（同一色相の濃淡のみ）は、実際にレビューした際に線が重なる箇所で見にくいとの指摘を受け、色相変更＋線幅の段階化＋フォーカス線のハロー（縁取り）という二次エンコーディングを追加する再設計が必要になった。

### 4. スタック構成のPRは、マージ直前に必ずbaseがmainかどうか確認する
- **内容**: 複数PRが連鎖するスタック構成では、あるPRのbaseがstaleな祖先ブランチのまま残っていると、内容が本来のPRを経由せず別ブランチへ混入する事故になる。
- **対応**: finish-reviewスキル手順2のbase整合性チェック（PR #49で追加）を必ず実施する。base元PRが未マージなら対象PRをマージしない。
- **実例**: PR #42が`base: docs/issue-29-class-diagram`のまま（本来`main`へ変更すべきところ）マージされ、Issue #29・#30の内容がPR #41・#42という本来のPRを経由せず、その子孫であるPR #43経由で`main`へ混入した（2026-07-15）。
